# Altitude Pro — SEO Foundation (Phases 1–3) Design

**Date:** 2026-04-21
**Author:** maxbrustein (with Claude)
**Status:** Approved — ready for implementation plan
**Scope:** First three phases of the SEO master plan (`altitude-pro-seo-plan.md`). Deferred: marketing homepage (Phase 4), auth/Supabase (Phase 5), content articles (Phase 6), monetization (Phase 7).

## Goal

Get Altitude Pro's study content indexed by Google as 40 separate static HTML pages, one per ACS task, under clean URLs like `https://altitudepro.org/study/pilot-qualifications`. Each page is fully crawlable with semantic headings, schema markup, canonical URLs, sitemap entry, and per-card anchor links that can surface as Google sitelinks.

Keep the existing SPA (Vite + vanilla JS + hash routing) fully working — it moves to `/app` but is otherwise untouched.

**Non-goals**:
- Marketing homepage copy (Phase 4 — out of scope)
- Login / Supabase auth (Phase 5 — out of scope)
- Per-task content rewrites for SEO (use existing task HTML as source)
- Blog-style articles (Phase 6)
- Any monetization work

## Core architectural collision (and resolution)

The existing app is a single-page Vite app with hash routing (`#/area/I/task/C`). Google does not index hash fragments, and all task content is injected into the DOM at runtime via JS bundling — the initial HTML contains no study content.

**Resolution**: generate 40 static HTML pages at build time from the same `manifest.json` + task HTML partials the SPA already uses. Pages live at `/study/<slug>`; the SPA moves to `/app`. Content source stays single; a new build script emits two artifacts (SPA bundle + static SEO pages).

## Decisions

| Decision | Choice | Alternatives considered |
|---|---|---|
| Architecture | Keep SPA, add static SEO pages alongside it | Migrate to Astro SSG; pre-render SPA via Vite plugin |
| Scope | Phases 1–3 only (infra + 40 task pages + Search Console) | Include Phases 4–5 (marketing + auth); include Phases 6–7 |
| Domain | `altitudepro.org` (user-owned) | Vercel preview URL (SEO-weaker); buy different domain |
| Content transform | Build-time script converts existing task HTML to semantic HTML | Reuse task HTML verbatim; hand-rewrite per page |
| URL slugs | Hybrid — canonical auto-slugged from title + optional `aliases[]` 301-redirecting to canonical | Auto only; manual explicit slugs; separate slug-map file |
| Standalone pages | Deferred (VFR minimums, METAR decoder, etc.) | Include all 7; include 2–3 easy ones |
| SPA location | `/app` — clean future-proof structure | Keep at `/` |
| Root path `/` | 302 redirect to `/app` during this phase | Bare placeholder page; keep SPA at `/` |
| Anchor links | Full per-doc: subsection + fact-card level, copy-to-clipboard JS, `:target` highlight | Subsection-only; skip entirely |
| Auth-gated CTAs | Point at `/app` for now; swap to `/signup` when auth ships later | Skip CTAs entirely |

## Architecture

### Target URL structure

```
altitudepro.org/
├── /                         → 302 redirect to /app (placeholder slot for Phase 4 marketing home)
├── /app                      → existing SPA (hash routes preserved internally)
├── /study/<slug>             → 40 static HTML pages, one per ACS task
├── /sitemap.xml              → generated by build
└── /robots.txt               → allows all crawlers
```

### Single source of truth

`content/certs/ppl/manifest.json` and `content/certs/ppl/tasks/<id>.html` stay authoritative. Both the SPA's runtime injection and the new prerender script read the same files. Editing a task updates both outputs.

### Build pipeline

```
npm run build
  ├─ node scripts/build-seo.js
  │    ├─ load manifest + task partials
  │    ├─ emit public/study/<slug>.html  (40 files)
  │    ├─ emit public/sitemap.xml
  │    ├─ emit public/robots.txt
  │    └─ emit/update vercel.json  (redirects + rewrites)
  └─ vite build
       └─ bundles SPA to dist/app/ + copies public/* to dist/
```

No new runtime dependencies. Prerender is pure Node + string templates + minimal HTML parsing.

## Content transform

### Input

Existing task partial, e.g., `content/certs/ppl/tasks/I-A.html`:

```html
<div class="task-section task-anchor" id="t-ia" data-section-val="I · PREFLIGHT PREP" data-task-val="A · PILOT QUALS">
  <div class="task-hdr">
    <span class="task-badge">Task A</span>
    <span class="task-title">Pilot Qualifications</span>
  </div>
  <div class="subsec-lbl">Certificate Requirements <span class="src-tag far">14 CFR 61</span></div>
  <div class="fact-grid two-col">
    <div class="fact-card hi">
      <div class="fact-title">Part 61 Hour Requirements</div>
      <div class="fact-body">...</div>
      <div class="fact-ref">14 CFR 61.109(a)</div>
    </div>
  </div>
</div>
```

### Transform rules

- `.task-title` text becomes the page's `<h1>`; extracted from content flow (the page template provides the h1 slot).
- `<div class="subsec-lbl">X <span class="src-tag">Y</span></div>` →
  `<h2 id="x-slug">X <span class="src-tag">Y</span> <a class="anchor-link" href="#x-slug">§</a></h2>`
- `<div class="fact-card"><div class="fact-title">T</div>...</div>` →
  `<section class="fact-card" id="t-slug"><h3>T <a class="anchor-link" href="#t-slug">§</a></h3>...</section>`
- `.fact-title` inner contents become the h3; `.fact-body` and `.fact-ref` contents unchanged.
- `linkifyRefs()` still applied — citations remain clickable.
- All existing CSS classes preserved — same stylesheet renders the study pages.

### Anchor ID generation

- Lowercase kebab-case from heading/card title text
- Strip punctuation; collapse whitespace to `-`
- Override table for common mnemonics/abbreviations:
  - "TOMATO FLAMES" → `tomato-flames`
  - "AROW — Must Be On Board" → `arow-required-documents`
  - "AVIATES — Required Inspection Mnemonic" → `aviates-inspections`
  - (Expand override table as needed; starts small, grows from real card titles)
- Regulation-specific card titles get regulation IDs: `#14-cfr-91-155`, `#14-cfr-61-109`
- Build fails if two elements on the same page share an ID

## URL + slug system

### Manifest schema extension

```json
{
  "id": "I-F",
  "letter": "F",
  "title": "Performance & Limitations",
  "short": "PERFORMANCE",
  "slug": "performance-and-limitations",
  "aliases": ["aircraft-performance", "weight-and-balance"],
  "references": ["POH/AFM", "FAA-H-8083-3"]
}
```

- `slug` — canonical URL. Auto-derived from `title` at build time if omitted, but pre-populated on every task in the first commit for explicitness.
- `aliases` — optional array. Each emits a 301 to the canonical.

### Auto-slug rules

- Lowercase
- `&` → `and`
- `/` → `-`
- Strip all other punctuation
- Spaces → `-`
- Collapse consecutive `-`

E.g., `"Performance & Limitations"` → `performance-and-limitations`.

### `vercel.json` redirects

Build script generates redirects block:

```json
{
  "redirects": [
    { "source": "/study/aircraft-performance", "destination": "/study/performance-and-limitations", "permanent": true },
    { "source": "/study/weight-and-balance",    "destination": "/study/performance-and-limitations", "permanent": true }
  ]
}
```

### Auto-generated intro paragraph

One-line template, populated per task from manifest fields:
> "Everything you need to know about **{title}** for your private pilot checkride. Aligned to **FAA-S-ACS-6C Task {id}**, covering {first-3-subsection-titles}."

Subsection titles are extracted from the transformed HTML during prerender.

### Styling

- Reuses the AeroForce CSS tokens (`:root { --void, --glow, ... }`)
- **`public/study/assets/study.css` is a new file**, hand-assembled from `src/styles/tokens.css` + the content-relevant rules in `src/styles/base.css` and `src/styles/components.css` (sidebar/layout/quiz rules excluded), plus new header/footer/breadcrumb/CTA/anchor-link rules specific to the study page. Served as a static asset — not bundled by Vite. Standalone so study pages don't pull in the app's JS or unused CSS.
- Risk: app styles and study styles can drift over time. Acceptable — study pages change rarely once shipped, and drift would manifest as visual-only differences spot-checked during content edits.
- Mobile-responsive; no horizontal scroll; touch targets ≥44px.
- Inherits the same `.fact-card`, `.subsec-lbl`, `.hl`, `.src-tag` styles readers see in the app.

## Anchor link system

### Behavior

- Every `<h2>` subsection has a unique `id` and a hover-revealed `§` link
- Every `.fact-card <h3>` has a unique `id` and a hover-revealed `§` link
- On hover of the card/heading, the `§` fades in
- Clicking `§` copies the full URL to clipboard + flashes `✓` briefly + updates browser URL via `history.pushState`
- Arriving via anchor (`/study/x#y`) triggers a `:target` CSS glow on the matched element — subtle green outline for ~1.5 seconds

### CSS

```css
.anchor-link { opacity: 0; margin-left: 8px; color: var(--glow); text-decoration: none; font-family: var(--fm); font-size: 11px; transition: opacity 0.15s; user-select: none; }
.fact-card:hover .anchor-link, h2:hover .anchor-link { opacity: 1; }
@media (hover: none) { .anchor-link { opacity: 0.4; } }
.fact-card:target { border-color: var(--glow); box-shadow: 0 0 0 1px var(--glow), 0 0 20px rgba(157,217,104,.15); transition: box-shadow 0.3s ease; }
h2:target { color: var(--neon); }
```

### JS (inline or in `/study/assets/study.js`)

```js
document.querySelectorAll('.anchor-link').forEach(link => {
  link.addEventListener('click', e => {
    e.preventDefault();
    const href = link.getAttribute('href');
    const url = window.location.origin + window.location.pathname + href;
    history.pushState(null, '', href);
    navigator.clipboard.writeText(url).then(() => {
      const orig = link.textContent;
      link.textContent = '✓';
      setTimeout(() => link.textContent = orig, 1200);
    });
  });
});
```

## SEO infrastructure

### `robots.txt`

```
User-agent: *
Allow: /
Sitemap: https://altitudepro.org/sitemap.xml
```

### `sitemap.xml`

Generated by the build script. Entries:
- `https://altitudepro.org/` (priority 1.0, no lastmod)
- `https://altitudepro.org/study/<slug>` for each of the 40 tasks (priority 0.8, changefreq monthly, lastmod = git log date of the task HTML file)

`/app` is intentionally **not** in the sitemap — the SPA shell has no indexable content, and we don't want to invite crawlers into the interactive app. The 302 at `/` plus backlinks from every study page's header are enough to let users and crawlers discover the app.

### `vercel.json`

Produced by the build script — never hand-edited for routing:

```json
{
  "rewrites": [
    { "source": "/app/:path*", "destination": "/app/index.html" }
  ],
  "redirects": [
    { "source": "/", "destination": "/app", "permanent": false },
    // one entry per alias
    { "source": "/study/aircraft-performance", "destination": "/study/performance-and-limitations", "permanent": true }
  ]
}
```

The `/` redirect is a 302 (temporary) so Google doesn't permanently map the root to `/app` — we want the marketing homepage at `/` eventually.

### Google Search Console

Manual one-time setup after first deploy (documented in plan):
1. Add property `altitudepro.org` at `search.google.com/search-console`
2. Verify via DNS TXT record (registrar-side action by user)
3. Submit `https://altitudepro.org/sitemap.xml`
4. Monitor Index Coverage report weekly for first month

### Analytics

`@vercel/analytics` is already loaded by the SPA via `src/main.js`. Study pages load the Vercel insights script directly:

```html
<script defer src="/_vercel/insights/script.js"></script>
```

Tracks page views + visitor counts per study page. No additional package install.

## Build + deploy

### Build command

```json
"scripts": {
  "build": "node scripts/build-seo.js && vite build"
}
```

### `scripts/build-seo.js` (one file, ~200 lines)

Responsibilities:
1. Load `content/certs/ppl/manifest.json`
2. For each task: read `content/certs/ppl/tasks/<id>.html`, transform to semantic HTML, inject into page template, write to `public/study/<slug>.html`
3. Validate:
   - Every task has a `slug`
   - All slugs unique across manifest
   - All aliases unique (and don't collide with any canonical slug)
   - Every generated page has unique anchor IDs within itself
4. Generate `public/sitemap.xml`
5. Generate `public/robots.txt`
6. Generate/update `vercel.json`
7. Report: count of pages emitted, aliases, anchors, any validation errors; exit non-zero on error

### Vite config adjustment

```js
// vite.config.js
export default defineConfig({
  base: '/app/',
  server: { port: 5173 },
  build: {
    outDir: 'dist/app',
    sourcemap: true
  }
});
```

Vite outputs to `dist/app/`; `public/` (containing `study/`, `sitemap.xml`, `robots.txt`) is copied to `dist/` root (Vercel serves `dist/` as the site root).

### Tests

New file `tests/build-seo.test.js` (~80 lines):
- Manifest has no duplicate slugs
- Manifest has no duplicate aliases; no alias collides with a slug
- Every task has slug, non-empty title
- Slug format matches `/^[a-z0-9-]+$/`
- After running the build script (in test mode, emits to a tmp dir): every study page file exists; every file has `<h1>`, canonical link, meta description, Article schema, at least one `<h2>`, at least one `.fact-card[id]`

All existing tests (content, state, linkify) continue to pass unchanged.

### First deploy + domain wiring

1. Merge to `main` → Vercel auto-deploys to `altitude-pro.vercel.app` with the new structure (SPA now at `/app`, study pages at `/study/*`, `/` redirects to `/app`)
2. In Vercel dashboard: add `altitudepro.org` as a domain on the project
3. At domain registrar: add DNS records per Vercel's instructions (A record for apex, CNAME for www, OR Vercel nameservers)
4. Wait for SSL provisioning (<5 min typically)
5. Set `altitudepro.org` as the production domain in Vercel
6. Register the property in Google Search Console, verify via DNS TXT
7. Submit sitemap
